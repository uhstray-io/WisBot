# WisBot → agent-cloud Deployment Alignment Plan

**Date:** 2026-06-01
**Status:** DRAFT — pending review
**Goal:** Make WisBot deployable through the uhstray-io `agent-cloud` platform's standard pattern (OpenBao → Ansible → templated `.env` → pulled image → Semaphore orchestration), and remove site-specific config/secrets from this repo.

---

## 1. Context

`agent-cloud` (`uhstray-io/agent-cloud`) is the platform monorepo and single source of truth for deployments. WisBot is intentionally **kept as a separate public repo** (independent .NET toolchain, own release cadence) and integrates as an **AI/Agent-tier** service — its deployment definition lives under **`agents/wisbot/`** in agent-cloud (where the other agents live), not `platform/services/`. The platform's contract for such a service:

- WisBot publishes a **prebuilt Docker image**; agent-cloud **pulls** it (no build-from-source on the VM).
- All config arrives via a **templated `.env`** file generated by Ansible from **OpenBao** secrets + **site-config** values. The container only *reads* env — it never fetches secrets itself.
- **No secrets, real IPs, real IDs, or usernames** live in either repo's committed files. Real values live in the private **site-config** repo and OpenBao (`secret/services/discord`).
- Deploys run **only through Semaphore** (`deploy-wisbot.yml`), never by SSHing and running `deploy.sh` directly.
- Health is verified over **HTTP** (`service_url` + `health_path`).

The reserved OpenBao path `secret/services/discord` (Discord bot token) and WisBot's AI/Agent-tier classification (full pattern + `context/` dir) already exist in agent-cloud docs.

## 2. Current State (gap analysis)

| Area | Current state in WisBot | Required for agent-cloud |
|------|------------------------|--------------------------|
| **Dockerfile** | **Missing** (CI calls `docker build .` with no Dockerfile) | Multi-stage .NET 10 Dockerfile producing a Linux image |
| **compose.yml** | **Missing** (CI calls `docker compose` with no file) | Lives in agent-cloud `agents/wisbot/deployment/` |
| **Voice native libs** | `OpusDotNet.opus.win-x64` + `copy-native-libs.bat` (Windows-only `.dll`s) | Linux-native opus + libsodium installed in the image |
| **Discord token** | `discord.key` file (gitignored) or `DISCORD_TOKEN_WISBOT` env; CI appends it and `cat`s `.env` (**leaks to logs**) | From OpenBao → Ansible → `.env`; never printed; never committed |
| **Guild/channel IDs** | Hardcoded in `Bot.cs`: `uhstrayGuildId=889910011113906186` (3 uses), `testGuildId`, `testChannelId=889910012019867672` | Env-configurable; real values in site-config |
| **Ollama endpoint** | Env-configurable (`OLLAMA_ENDPOINT`), good. `.env.example` uses placeholder IP | Keep as config; prod value (WisAI/Open WebUI) set in deployment |
| **DB path** | Hardcoded relative `wisbot.db` (`Database.cs`) | Env-configurable + mounted volume |
| **Recordings path** | Hardcoded `GetCurrentDirectory()/recordings` (`VoiceRecorder.cs`) | Env-configurable + mounted volume (MinIO later) |
| **Health check** | None over HTTP (only Discord `/status` command) | Minimal HTTP `/health` endpoint on a configurable port |
| **Deploy workflow** | Self-hosted runner runs `docker build` + `docker compose up` | CI only **builds + publishes image**; deploy moves to Semaphore |

## 3. Decisions (locked)

1. **Scope:** both repos — WisBot changes (primary) + companion `agents/wisbot/` in agent-cloud.
2. **Voice on Linux:** swap to Linux-native opus + libsodium so voice recording keeps working in the container.
3. **Image registry:** GitHub Container Registry — `ghcr.io/uhstray-io/wisbot` (public repo, native to GitHub Actions). *(Recommended default; swap to Harbor later if the platform standardizes on it.)*
4. **Plan location:** this file, `docs/plans/`.

## 4. Workstreams

### WS1 — Containerization (WisBot repo)

- [ ] Add a multi-stage **`Dockerfile`** (Linux):
  - Build stage: `mcr.microsoft.com/dotnet/sdk:10.0` → `dotnet restore` + `dotnet publish -c Release -r linux-x64` (or framework-dependent).
  - Runtime stage: `mcr.microsoft.com/dotnet/runtime:10.0` (console app, not aspnet unless the health endpoint uses Kestrel — see WS3).
  - `apt-get install -y libopus0 libsodium23` (Linux native voice libs) in the runtime stage.
  - Run as non-root user; set `WORKDIR /app`; `ENTRYPOINT ["dotnet","Wisbot.dll"]`.
- [ ] Add **`.dockerignore`** (`bin/`, `obj/`, `.git/`, `*.key`, `.env`, `recordings/`, `*.db`).
- [ ] **`Wisbot.csproj`:** remove `OpusDotNet.opus.win-x64`; make voice libs OS-agnostic.
  - Replace with a cross-platform Opus binding **or** rely on the apt-installed `libopus0`/`libsodium23` with a `DllImportResolver` / symlinks so Discord.Net loads `opus`/`libsodium` by the names it expects.
  - **Verification step (spike):** confirm Discord.Net 3.19-beta voice connect + `GetStreams()` works on Linux with the apt libs in a throwaway container before finalizing. Document the exact `.so` names Discord.Net probes.
- [ ] Delete or Windows-gate `copy-native-libs.bat` (it's a local-dev Windows helper; keep it out of the Linux build path).

### WS3 — HTTP health endpoint (WisBot repo)

agent-cloud's Ansible verify hits `service_url + health_path`. WisBot has no HTTP server.

- [ ] Add a minimal HTTP listener (lightweight `HttpListener`, or `WebApplication` on Kestrel if we move to the aspnet runtime image) bound to `0.0.0.0:${WISBOT_HEALTH_PORT}`.
- [ ] `GET /health` → `200` when the Discord client is connected (`ConnectionState.Connected` / `client.Login` done), `503` otherwise. Body: small JSON (uptime, latency, guild count) — reuse the data `StatusService` already computes.
- [ ] Start it from `Program.cs` alongside the bot/terminal tasks.
- [ ] Add a compose-level healthcheck (WS5) that curls it.

### WS5 — Companion deployment in agent-cloud (`agents/wisbot/`)

> **Corrected after deeper review (2026-06-01):** WisBot is an **AI agent** → it lives under **`agents/wisbot/`**, NOT `platform/services/wisbot/` (verified: nemoclaw/netclaw/cowork/websmith all live under `agents/<name>/` with the `deployment/ + context/` split; their deploy playbooks still sit at `platform/playbooks/deploy-<name>.yml`). Two deploy precedents exist: **agents like nemoclaw clone the external source repo and build via its installer** (no compose, no prebuilt image); **platform services like n8n/nocodb pull a prebuilt image + compose**. Because WisBot **publishes its own image to GHCR**, it uses the **pull-image + compose** mechanics *inside* the `agents/` directory — a hybrid that's cleaner than nemoclaw's clone-and-build.

Follow `plan/architecture/SERVICE-INTEGRATION-PLAN.md` (AI/Agent tier → full pattern + `context/`) and the composable pattern in `AUTOMATION-COMPOSABILITY.md` (source/runtime-dir split, Jinja2 + `manage-secrets.yml`, NetBox is the reference impl).

- [ ] `agents/wisbot/deployment/` (mirror nemoclaw's file set):
  - **`compose.yml`** — pull `ghcr.io/uhstray-io/wisbot:${WISBOT_IMAGE_TAG:-latest}`; `env_file: ./env/wisbot.env`; named volumes for DB + recordings; `restart: always`; healthcheck `curl -sf http://localhost:8080/health`; bind via `${WISBOT_LISTEN:-127.0.0.1}` (internal only — **no Caddy route**, confirmed: Caddy is public-web only; WisBot is outbound-to-Discord + internal `/health`).
  - **`deploy.sh`** — **container-lifecycle only**, and must satisfy CI: `#!/usr/bin/env bash`, executable, source `${CLONE_DIR}/platform/lib/common.sh`, use `$CONTAINER_ENGINE` (never hardcode docker/podman), `compose pull` + `compose up -d`, **fail immediately if `.env`/`env/wisbot.env` missing** (means Ansible didn't run), `wait_for_http`. **No** `gen_secret`/`get_secret`/OpenBao calls. ShellCheck-clean (no SC2086/SC2064/SC2155).
  - **`templates/wisbot.env.j2`** — `DISCORD_TOKEN_WISBOT={{ secrets.discord_token }}`, `OLLAMA_ENDPOINT={{ wisbot_ollama_endpoint }}`, `WISBOT_GUILD_ID={{ wisbot_guild_id }}`, model/context tunables, paths. **Secrets via `{{ secrets.* }}`; site values via `{{ inventory_var }}` — never literal** (CI greps `.j2` for hardcoded IPs/creds).

  - **`CLAUDE.md`** + **`README.md`** + **`.env.example`** (mirror nemoclaw/netbox structure).
  - **`context/`** — `architecture/`, `prompts/`, `skills/`, `use-cases/` (AI-tier requirement; `.gitkeep` to start).
- [ ] `platform/playbooks/deploy-wisbot.yml` — composable (model on `deploy-netbox.yml`): `sparse-checkout` → `manage-secrets` (renders `wisbot.env.j2` → runtime dir) → `run-deploy` → `verify-health`. Define `_secret_definitions` (`discord_token` = **`user`** type — operator-managed, never auto-generated) and `_env_templates`. *(Thin-wrapper `import_playbook: deploy-service.yml` is the simpler legacy alternative nemoclaw uses — acceptable but less aligned with the stated standard.)*

- [ ] Add Semaphore templates to `platform/semaphore/templates.yml` (Deploy WisBot / Clean Deploy WisBot, `service_branch` survey var); run `setup-templates.yml`.

## 5. Persistence

- **SQLite** (`wisbot.db`) → named volume mounted at the configurable `WISBOT_DB_PATH` (e.g. `/app/data/wisbot.db`). Survives restarts/redeploys.
- **Recordings** → named volume at `WISBOT_RECORDINGS_DIR` (e.g. `/app/recordings`).
- **Future (out of scope, noted in agent-cloud as a gap):** SQLite → PostgreSQL for prod HA; recordings → MinIO bucket `wisbot-recordings/`. Track as a follow-up plan.

## 6. Risks & open questions

1. **Voice on Linux (highest risk).** Discord.Net 3.19-beta voice depends on native opus/libsodium by specific probe names. The apt packages provide the `.so`s, but a `DllImportResolver` or symlinks may be needed. **Mitigation:** the WS1 spike validates this in a container before we commit. If it can't be made to work quickly, fall back to "voice disabled in container, re-enable in follow-up" (everything else — LLM chat, reminders, welcome, stats — is unaffected).
2. **Health endpoint runtime image.** `HttpListener` keeps us on the smaller `runtime` image; Kestrel needs `aspnet`. Recommend `HttpListener` to avoid pulling ASP.NET unless we want richer endpoints.
3. **site-config access** — needed for WS6 only (real IPs/IDs, token seeding). WS1–WS5b don't need it. **Please grant access when we reach WS6**, or hand me the guild ID / WisAI endpoint to slot into the inventory template.
4. **OpenBao path choice** (`secret/services/wisbot` recommended vs reserved `secret/services/discord`) — one-line decision (WS5).
5. **o11y / shutdown workflows** — confirm whether observability + shutdown automation should migrate to agent-cloud rather than stay as WisBot self-hosted workflows.
6. **Architecture mismatch** — build `linux-amd64`; if any target VM is arm64, add multi-arch buildx.
7. **Composable vs thin-wrapper playbook** — recommend the full composable `deploy-wisbot.yml` (NetBox-style, `manage-secrets` + Jinja2) over nemoclaw's thin `deploy-service.yml` wrapper, since it's the documented standard and cleanly templates the token. Confirm preference.

**Resolved by the 2026-06-01 deep review:** placement is `agents/wisbot/` (not `platform/services/`); **WisBot needs no Caddy route** (internal `/health` only); **no runtime OpenBao AppRole** (deploy-time token only); CI gates require **BATS tests + no committed `.env` + no hardcoded IPs/creds + mermaid-only diagrams** (see WS5b).

## 7. Phased Execution

**Workflow per phase (mandatory):** branch from `main` → implement → `dotnet build` green → push → open PR → **wait for CodeRabbit + CI** → address findings → confirm checks pass → merge → delete branch. One phase = one branch = one PR. No phase starts until the prior phase is merged (phases are ordered to avoid conflicts). No `Co-Authored-By` trailers (per repo memory).

Phases are ordered **lowest-risk / prerequisite first**, so each PR is small, independently reviewable, and leaves `main` shippable.
